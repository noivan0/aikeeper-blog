## 🔍 Cursor AI가 API 문서를 만드는 원리, 먼저 알고 가야 합니다

비개발자 입장에서 Cursor AI를 이해할 때 가장 중요한 개념이 하나 있어요. 바로 **"컨텍스트(Context)"** 개념입니다.

### Cursor가 문서를 생성하는 방식

Cursor AI는 단순히 "코드를 설명해줘"라고 하면 작동하는 도구가 아닙니다. 핵심은 **@파일명** 기능이에요. Cursor의 Chat 패널에서 `@routes/user.py` 처럼 파일을 직접 지정하면, AI가 그 파일 전체를 읽고 "이 코드가 무엇을 하는지"를 파악한 다음 문서를 생성합니다.

2026년 4월 기준으로 Cursor AI는 Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro 등 멀티 LLM을 지원하고, 코드베이스 전체를 인덱싱해서 참조합니다. 즉, 단일 파일이 아니라 "프로젝트 전체의 맥락"을 이해한 상태에서 문서를 만들 수 있어요.

제가 테스트한 프로젝트는 Flask 기반의 REST API였습니다. 엔드포인트가 총 34개였고, 각각 파라미터 3~8개, 응답 코드 4~6가지를 가진 중간 규모 프로젝트였어요. 이 규모의 API 문서를 수동으로 작성하면 개발자 기준으로 하루 반, 저 같은 비개발자 기준으로 꼬박 3~4일 걸립니다.

### 실제로 비개발자가 쓸 수 있는 두 가지 진입 방법

**방법 1 – Chat 패널 활용 (추천)**
Cursor를 열고 오른쪽 Chat 패널에 아래처럼 입력합니다.

```
@routes/user.py 이 파일에 있는 모든 API 엔드포인트를 Markdown 형식으로 문서화해줘.
각 엔드포인트마다 다음을 포함해줘:
- HTTP 메서드와 URL
- 요청 파라미터 (이름, 타입, 필수 여부, 설명)
- 응답 예시 (성공/실패)
- 에러 코드
```

이렇게 하면 Cursor가 코드를 읽고 Markdown 문서를 뽑아줍니다. 저는 이 방법으로 34개 엔드포인트 중 28개의 초안을 **총 2시간 40분**만에 생성했어요.

**방법 2 – Composer 기능으로 대규모 일괄 처리**
여러 파일을 한꺼번에 처리할 때는 Composer(Cmd+I)를 씁니다. 여러 @파일을 한 번에 지정해서 "전체 API 문서를 한 파일로 통합해줘"라고 요청할 수 있어요.

> 💡 **실전 팁**: 한 번에 너무 많은 파일을 넣으면 컨텍스트 초과로 품질이 떨어집니다. 파일 5~7개씩 묶어서 처리하고, 나중에 합치는 방식이 더 정확도가 높아요.

### 1주차: 진입 장벽과 세 번의 실패

**Day 1~2**: 설치는 쉬웠습니다. Cursor 홈페이지에서 다운로드, VSCode 기반이라 생김새도 익숙해요. 문제는 코드를 열었을 때였어요. 어떤 파일이 어떤 역할인지 전혀 모르는 상태에서 Chat에 "API 문서 만들어줘"라고 치면... 당연히 엉터리 결과가 나옵니다.

**첫 번째 실패**: 파일 구조를 이해하지 못한 채로 최상위 폴더 전체를 @로 지정했더니, Cursor가 설정 파일, 데이터베이스 마이그레이션 파일까지 전부 섞어서 읽고 이상한 "문서"를 뱉었습니다.

**Day 3**: 개발자에게 30분짜리 "코드 투어"를 요청했어요. "이 프로젝트에서 API 엔드포인트가 있는 파일은 어디에 있어요?"라고만 물었습니다. `routes/` 폴더 안에 있다는 걸 알고 나서 작업 효율이 완전히 달라졌어요.

**Day 4~5**: `routes/` 하위 파일들을 하나씩 처리하기 시작했어요. user.py, product.py, order.py 순으로 Chat 패널에 넣으니, 각 파일당 10~20분 만에 초안이 나왔습니다.

**두 번째 실패**: 무료 플랜 한도(월 50회) 소진. 5일 만에 고급 모델 사용 횟수를 다 쓰고 GPT-3.5로 다운그레이드됐어요. 품질 차이가 확연했습니다. 파라미터 설명이 뭉뚱그려지고, 에러 코드가 빠지는 경우가 많아졌어요.

**Day 6**: Pro 플랜($20/월)으로 전환. [Cursor 공식 사이트에서 가격 확인](https://www.cursor.com/pricing)

### 2주차: 본격 활용과 진짜 한계 발견

**Day 8~10**: 리듬을 찾았습니다. 아침에 파일 5개씩 Chat에 넣고 초안을 뽑은 뒤, 오후에 개발자 슬랙 채널에 "이 에러 코드 설명이 맞나요?"라고 확인 요청하는 루틴이 자리잡혔어요.

**Day 11**: 세 번째 실패이자 가장 중요한 한계를 발견했습니다. Cursor가 생성한 문서에 **비즈니스 맥락이 전혀 없었어요**. "이 API는 왜 존재하는지", "어느 팀이 소비하는지", "언제 쓰면 안 되는지" 같은 내용은 코드에 없으니 AI도 모릅니다. 이 부분은 결국 제가 직접 써야 했어요.

**Day 14**: 최종 결과물 정리. 34개 엔드포인트 문서화 완료까지 총 소요 시간 **약 7시간**. 과거 수동 작업 대비 약 18~20시간에서 7시간으로 줄었으니, **시간 절감률 약 65%**였습니다.

> 💡 **실전 팁**: Cursor가 생성한 문서를 Notion이나 Confluence에 붙여넣기 전에, 반드시 "Overview" 섹션(이 API의 목적, 사용 대상, 주의사항)을 직접 추가하세요. 이 부분이 없으면 외부 팀이 쓸 수 없는 문서가 됩니다.

### ChatGPT로 그냥 하면 안 되나요?

이 질문을 많이 받아요. 결론부터 말하면 **소규모로는 ChatGPT도 충분하지만, 실무 규모에서는 한계가 명확합니다**.

반면 Cursor는 프로젝트 전체를 인덱싱해서, `user.py`의 함수가 `models/user.py`의 클래스를 참조한다는 걸 알고 문서를 작성합니다. 이 차이가 실제 문서 완성도에서 크게 납니다.

> 💡 **실전 팁**: 예산이 없다면 ChatGPT + 수동 복붙으로 소규모 API 문서화는 가능합니다. 그러나 10개 이상 엔드포인트가 있는 프로젝트라면 Cursor Pro가 시간 대비 효율이 훨씬 높아요.

## 🔍 비개발자가 빠지기 쉬운 함정 5가지

2주 동안 제가 직접 겪고, 다른 사례에서 수집한 실수들입니다. 이것만 안 해도 시간을 절반은 아낄 수 있어요.

### 함정 1: 파일 구조 파악 없이 바로 시작하기

가장 흔한 실수입니다. "AI니까 알아서 하겠지"라고 생각하고 루트 폴더 전체를 던지면, Cursor는 관련 없는 설정 파일, 테스트 파일, 마이그레이션 파일까지 섞어서 읽습니다. **반드시 개발자에게 "API 로직이 있는 파일 위치"를 먼저 확인하세요.** 30분 투자가 3시간 삽질을 막습니다.

### 함정 2: 한 번에 너무 많은 파일 처리

Cursor의 컨텍스트 창에는 한계가 있어요. 한 번에 파일 10개 이상을 @로 지정하면 뒤쪽 파일은 제대로 읽히지 않습니다. **파일 5~7개씩 배치로 처리하고 결과물을 합치는 방식**이 품질이 훨씬 안정적입니다.

### 함정 3: AI 출력을 그대로 믿기

Cursor가 생성한 파라미터 설명이 "그럴싸하게" 틀릴 수 있어요. 특히 `required: true`인지 `optional`인지, 기본값이 뭔지, 허용 값의 범위는 뭔지 같은 세부 사항은 반드시 개발자에게 확인받아야 합니다. **"AI 초안 → 개발자 15분 검수"** 프로세스를 생략하면 외부에 잘못된 문서가 나갑니다.

### 함정 4: 비즈니스 컨텍스트 기대하기

앞서도 언급했지만, "이 API는 결제 실패 시 3번 재시도하고, 그래도 실패하면 CRM팀에 알림이 간다"같은 비즈니스 규칙은 코드 어디에도 명시적으로 안 나옵니다. AI는 이걸 모르고, 알 수도 없어요. **Overview 섹션과 사용 시나리오는 반드시 직접 작성해야 합니다.**

### 함정 5: 문서화를 "한 번만" 이벤트로 보기

Cursor로 문서를 만들고 "완성!"이라고 생각하는 순간, 다음 스프린트에 코드가 바뀌면 다시 뒤처집니다. **Cursor를 "문서 생성기"가 아니라 "문서 유지보수 파트너"로 보고, 코드 변경 시마다 업데이트하는 습관**을 만들어야 해요. 이걸 루틴화하지 않으면 한 달 뒤 또 처음부터 시작하게 됩니다.

**Q2: Cursor AI와 GitHub Copilot 차이가 뭔가요? 문서 생성에는 어느 게 더 낫나요?**

두 도구는 방향이 다릅니다. GitHub Copilot은 코드 작성 중 자동 완성에 최적화되어 있고, Cursor AI는 코드베이스 전체를 컨텍스트로 잡아 "이 코드가 무슨 역할을 하는지" 설명하고 문서를 생성하는 데 강점이 있어요. API 문서 자동화 관점에서는 Cursor가 훨씬 강력합니다. @코드파일 참조 기능으로 엔드포인트 함수를 직접 읽어서 파라미터 설명, 응답 예시, 에러 코드까지 한 번에 뽑아줍니다. Copilot은 이런 "전체 코드베이스 기반 문서 생성"이 상대적으로 약해요. 비개발자라면 Cursor의 Chat 패널 UI가 더 친숙하고 접근하기도 쉽습니다.

**Q3: 개발을 전혀 모르는 사람도 Cursor AI로 API 문서를 만들 수 있나요?**

완전히 혼자서 "처음부터 끝까지" 만들기는 어렵습니다. 최소한 코드를 열어서 어느 파일이 어떤 기능인지 개발자에게 한 번 설명 들을 필요가 있어요. 그 이후부터는 비개발자도 충분히 활용 가능합니다. 실제로 저도 백엔드 개발자에게 30분짜리 코드 투어를 받은 뒤, Cursor Chat에 "이 함수의 파라미터와 응답 형식을 Markdown으로 정리해줘"라고 입력해서 초안을 뽑았어요. 완성도는 개발자가 직접 작성한 문서의 70~80% 수준이었고, 나머지 20~30%는 제가 직접 확인하며 수정했습니다. 비개발자에게도 충분히 실용적인 도구예요.
