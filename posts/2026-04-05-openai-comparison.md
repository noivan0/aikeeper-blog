## 🔍 임베딩 모델이 RAG 품질의 80%를 결정하는 이유

많은 분들이 RAG 시스템의 성능은 LLM(Large Language Model, 대형 언어 모델)에 달려 있다고 생각합니다. GPT-4o를 쓰면 더 좋은 답이 나올 거라고요. 하지만 현장에서 RAG를 운영해본 엔지니어들은 다른 이야기를 합니다.

"Retrieval이 틀리면 Generation도 틀린다."

### RAG에서 임베딩의 역할: 검색 품질의 천장

RAG(Retrieval-Augmented Generation)의 작동 방식을 간단히 복기하면, 사용자 쿼리 → 임베딩 변환 → 벡터 DB에서 유사 문서 검색 → LLM에 컨텍스트로 전달 → 최종 답변 생성 순서입니다.

여기서 임베딩 모델이 하는 일은 "의미적으로 비슷한 텍스트를 수학적으로 가까운 벡터로 표현"하는 것입니다. 이 변환이 정밀하지 않으면, 쿼리와 실제로 관련 있는 문서를 찾지 못하고 엉뚱한 문서를 가져와서 LLM에 전달하게 됩니다. 아무리 GPT-4o가 똑똑해도 잘못된 컨텍스트로는 좋은 답을 낼 수 없죠.

2025년 Pinecone이 발표한 RAG 성능 분석 리포트에 따르면, [프로덕션 RAG 시스템에서 검색 단계(Retrieval)의 품질이 최종 답변 품질에 미치는 영향은 약 70~80%](https://www.pinecone.io/learn/rag/)에 달합니다. 임베딩 모델 교체만으로 전체 RAG 시스템 응답 정확도가 15~25% 개선된 사례도 다수 보고되고 있습니다.

### 임베딩 모델을 바꾸는 게 왜 이렇게 까다로운가

임베딩 모델 선택이 중요한 또 다른 이유는 **교체 비용**입니다. 한번 특정 임베딩 모델로 벡터 DB를 구축하고 나면, 모델을 바꾸려면 전체 문서 코퍼스를 처음부터 다시 임베딩해야 합니다. 벡터 공간 자체가 달라지기 때문입니다.

100만 개 청크(chunk), 청크당 평균 512토큰이면 약 5억 토큰의 재임베딩 비용이 발생합니다. OpenAI text-embedding-3-large 기준이면 약 $65, text-embedding-3-small이면 $10. 금액만 보면 작아 보이지만, 여기에 엔지니어 공수, 서비스 중단 리스크, 다운스트림 시스템 재검증 비용까지 더하면 이야기가 달라집니다.

> 💡 **실전 팁**: 프로젝트 초기 파일럿 단계에서 최소 2~3개 임베딩 모델로 소규모 A/B 테스트를 먼저 진행하세요. 이때 드는 비용(수 달러 수준)이, 프로덕션 이후 모델 교체 비용(수십~수백 배)을 막아줍니다.

## 🔍 선택 기준 1: 순수 검색 성능 — MTEB 벤치마크 그 이상

임베딩 모델 비교에서 가장 먼저 확인하는 지표는 [MTEB(Massive Text Embedding Benchmark)](https://huggingface.co/spaces/mteb/leaderboard)입니다. 2026년 4월 기준 주요 모델 성능을 정리하면 다음과 같습니다.

| 모델 | MTEB 평균 점수 | 차원수 | 컨텍스트 길이 | 최대 처리 |
|------|--------------|--------|------------|---------|
| OpenAI text-embedding-3-large | 64.6 | 3,072 (축소 가능) | 8,191 토큰 | 배치 처리 |
| OpenAI text-embedding-3-small | 62.3 | 1,536 (축소 가능) | 8,191 토큰 | 배치 처리 |
| Cohere Embed v4 | 65.1 | 1,024 | 128,000 토큰 | 배치/스트리밍 |
| Cohere Embed Multilingual v3 | 64.0 | 1,024 | 512 토큰 | 배치 처리 |
| BAAI/bge-m3 (오픈소스) | 63.6 | 1,024 | 8,192 토큰 | 자체 서버 |

### 벤치마크 1위가 내 RAG에서도 1위가 아닌 이유

MTEB는 57개 데이터셋, 8개 태스크 유형을 아우르는 범용 벤치마크입니다. 문제는 여기에 의료 전문용어, 법률 문서, 국내 금융 규정집 같은 한국어 도메인 특화 텍스트는 거의 없다는 점입니다.

실제 운영 환경에서 Cohere Embed v4가 OpenAI text-embedding-3-large보다 MTEB 점수는 0.5점 높더라도, 특정 법률 문서 검색 태스크에서는 반대 결과가 나오는 경우도 있습니다. 반대로 한국어가 포함된 멀티링구얼 평가에서는 Cohere Multilingual 모델이 OpenAI 대비 평균 8~12% 높은 Recall@5를 보이는 경우가 다수입니다.

**중요한 차이점 하나**: Cohere Embed v4는 최대 128,000 토큰의 컨텍스트 길이를 지원합니다. OpenAI embedding의 8,191 토큰 대비 약 16배입니다. 이 차이는 긴 법률 계약서, 논문, 재무 보고서를 청킹(chunking, 문서 분할) 없이 통째로 임베딩하거나 매우 긴 청크를 구성할 때 결정적으로 달라집니다.

> 💡 **실전 팁**: 도메인 특화 문서로 직접 Precision@5, Recall@5, MRR(Mean Reciprocal Rank, 평균 역순위)을 측정하세요. 최소 쿼리 50개, 문서 200개면 의미 있는 비교가 가능합니다. [RAGAS 라이브러리](https://docs.ragas.io/)를 활용하면 RAG 전용 자동 평가 파이프라인을 빠르게 구축할 수 있습니다.

## 🔍 선택 기준 2: 비용 구조 — 보이는 가격과 숨어 있는 비용

"어차피 임베딩은 저렴하잖아요"라고 생각하기 쉽습니다. 토큰당 단가만 보면 그렇습니다. 그런데 프로덕션 환경에서 임베딩 비용이 예상보다 10배 나오는 상황이 실제로 발생합니다.

### 숨겨진 비용 3가지

**① 초기 임베딩 비용 (One-time)**: 기존 문서 코퍼스를 벡터 DB에 넣는 데 드는 비용입니다. 1,000만 토큰짜리 문서셋이면 text-embedding-3-large 기준 $1.30, small 기준 $0.20입니다. 이건 상대적으로 작습니다.

**② 쿼리 임베딩 비용 (Ongoing)**: 사용자가 검색할 때마다 쿼리도 임베딩해야 합니다. DAU(일별 활성 사용자) 1만 명, 1인당 평균 5회 쿼리, 쿼리 평균 50 토큰이면 하루 250만 토큰, 한 달이면 75,000만 토큰(750M 토큰). text-embedding-3-large로는 약 $97.50/월. 생각보다 크죠?

**③ 벡터 DB 운영 비용**: Pinecone, Weaviate, Qdrant 등 벡터 DB도 별도 비용이 발생합니다. 고차원 벡터(3,072차원 vs 1,024차원)는 저장 비용과 검색 속도에서도 차이가 납니다. OpenAI text-embedding-3-large의 3,072차원은 Cohere의 1,024차원 대비 3배 저장 공간을 차지합니다.

단, OpenAI는 **Matryoshka Representation Learning(MRL)** 기법을 적용해 차원을 256이나 512로 줄여도 성능 손실이 최소화되는 기능을 제공합니다. 이를 활용하면 비용과 성능을 동시에 잡을 수 있습니다.

> 💡 **실전 팁**: 쿼리 볼륨이 많은 서비스라면 text-embedding-3-large보다 text-embedding-3-small + 차원 축소 조합이 비용 대비 성능 측면에서 더 합리적일 수 있습니다. 실제로 small 모델의 256차원 버전이 large 모델의 full 차원보다 특정 태스크에서 동등한 성능을 내는 케이스가 보고되고 있습니다.

## 🔍 선택 기준 3: 한국어·다국어 지원 — 이 기준이 모든 것을 바꾼다

국내 기업이 RAG를 구축할 때 가장 많이 놓치는 기준이 바로 이것입니다. "어차피 좋은 모델이면 한국어도 잘 되겠지"라는 가정이 실제 운영에서 무너집니다.

### OpenAI 임베딩의 한국어 현실

OpenAI text-embedding-3 시리즈는 한국어를 지원하고, 영어보다는 성능이 낮지만 실용적인 수준의 한국어 임베딩을 제공합니다. 그러나 몇 가지 현실적 제약이 있습니다.

첫째, 한국어 토크나이징(tokenizing, 텍스트를 토큰으로 쪼개는 과정) 비효율성입니다. OpenAI의 cl100k_base 토크나이저는 한국어를 영어 대비 약 2.5~3배 많은 토큰으로 분할합니다. 즉, 같은 의미의 텍스트를 임베딩하는 데 한국어가 영어보다 2~3배 비용이 더 든다는 뜻입니다.

둘째, 한국어 전용 벤치마크(KoSTS, KLUE-STS)에서 OpenAI 임베딩의 성능은 MTEB 영어 점수 대비 약 10~20% 낮게 측정되는 경향이 있습니다. 특히 의미가 비슷하지만 표현이 다른 한국어 문장 쌍(예: "가격 문의합니다" vs "비용이 얼마인가요?")의 유사도 계산에서 영어보다 정밀도가 떨어집니다.

### Cohere 멀티링구얼 임베딩의 강점

Cohere Embed Multilingual v3와 Embed v4는 100개 이상 언어를 전용 학습 데이터로 커버합니다. 한국어, 일본어, 아랍어 같은 비라틴 문자 언어에서도 영어에 준하는 임베딩 품질을 목표로 설계되었습니다.

특히 Cohere의 멀티링구얼 임베딩은 **크로스링구얼(cross-lingual) 검색**을 지원합니다. 한국어로 쿼리를 날려도 영어로 된 문서를 정확하게 찾아오는 기능입니다. 한국어-영어가 혼재된 기업 문서 환경(예: 영어 기술 문서 + 한국어 내부 보고서)에서 단일 임베딩 모델로 처리할 수 있다는 것이 큰 장점입니다.

| 항목 | OpenAI text-embedding-3 | Cohere Embed v4 Multilingual |
|------|------------------------|------------------------------|
| 지원 언어 수 | 다국어 (공식 수 미공개) | 100개 이상 (공식) |
| 한국어 토큰 효율 | 낮음 (영어 대비 2.5~3배) | 높음 (전용 토크나이저) |
| 크로스링구얼 검색 | 제한적 | 지원 |
| 한국어 벤치마크 성능 | MTEB 영어 대비 10~20% 낮음 | 영어에 근접 |
| 한국어+영어 혼합 문서 | 가능하나 비효율 | 권장 |

> 💡 **실전 팁**: 한국어 RAG를 구축 중이라면 오픈소스 옵션도 검토하세요. `intfloat/multilingual-e5-large-instruct`와 `BAAI/bge-m3`는 한국어 임베딩에서 상용 모델과 경쟁하는 성능을 보여주며, 자체 서버에서 운영하면 API 비용이 0입니다. 초기 GPU 서버 비용이 부담이라면 소규모에서는 Cohere 멀티링구얼, 대규모로 가면 자체 오픈소스 모델 전환도 좋은 전략입니다.

## 🔍 선택 기준 4: 보안·컴플라이언스 — 기업 RAG의 숨겨진 1순위 기준

스타트업이나 개인 프로젝트에서는 크게 신경 쓰지 않지만, 금융·의료·공공·대기업 환경에서는 이 기준이 나머지 셋을 합친 것보다 중요한 경우가 있습니다.

### API 방식 임베딩의 데이터 보안 리스크

OpenAI와 Cohere 모두 API 방식으로 제공되는 기본 임베딩 서비스를 사용하면, 여러분의 문서 텍스트가 외부 서버로 전송됩니다. OpenAI는 API 사용 시 입력 데이터를 모델 학습에 사용하지 않는다고 명시하고 있으며([OpenAI Privacy Policy](https://openai.com/policies/privacy-policy)), Cohere도 동일한 정책을 가지고 있습니다.

그러나 많은 국내 대기업, 금융기관, 공공기관에서는 사내 기밀 문서가 외부 API 서버에 전송되는 것 자체를 금지하는 정보보안 정책을 운영합니다. "데이터를 학습에 안 쓴다"는 보장만으로는 보안 심의를 통과하지 못하는 경우가 많습니다.

### 온프레미스·프라이빗 배포 옵션 비교

이 지점에서 OpenAI와 Cohere의 선택지가 갈립니다.

**OpenAI**: 현재 공식적인 온프레미스(on-premise, 자사 서버 직접 운영) 임베딩 배포 옵션이 없습니다. Azure OpenAI Service를 통해 특정 지역(Region)의 Azure 데이터센터에서 처리하도록 구성할 수 있어, 데이터가 특정 국가의 Azure 서버 내에서 처리되도록 제한할 수 있습니다.

**Cohere**: Cohere Private Deployment를 통해 고객사의 전용 클라우드 환경(AWS, GCP, Azure)이나 온프레미스 서버에 임베딩 모델을 직접 배포할 수 있습니다. 이 경우 데이터가 Cohere 서버로 나가지 않습니다.

**오픈소스(BAAI/bge-m3, multilingual-e5 등)**: 완전한 로컬 실행이 가능합니다. 데이터가 외부로 전혀 나가지 않습니다. 단, GPU 서버 구축·운영 비용과 모델 관리 역량이 필요합니다.

| 배포 방식 | OpenAI | Cohere | 오픈소스 |
|-----------|--------|--------|---------|
| 퍼블릭 API | ✅ | ✅ | ❌ |
| 특정 지역 처리 | Azure OpenAI | ✅ | N/A |
| 전용 클라우드 | ❌ | ✅ (별도 계약) | ✅ |
| 온프레미스 배포 | ❌ | ✅ (별도 계약) | ✅ |
| 데이터 완전 격리 | ❌ | ✅ (프라이빗) | ✅ |
| 비용 | API 종량제 | 별도 견적 | 서버 비용만 |

> 💡 **실전 팁**: 사내 정보보안팀과 임베딩 모델 선택 논의를 초기에 함께 하세요. 개발 완료 후 "이 API 사용하면 데이터 외부 전송이라 안 됩니다"라는 피드백을 받으면 아키텍처 전체를 뒤집어야 하는 상황이 옵니다. 컴플라이언스 제약이 있는 환경이라면 처음부터 Cohere Private Deployment나 오픈소스 모델을 설계에 포함하세요.

## 🔍 실제 사례: 기업들은 어떻게 임베딩 모델을 선택했나

### 사례 1: 네이버 클라우드 기반 법률 AI 스타트업 (익명, 2025년)

서울 소재 법률 AI 스타트업 A사는 초기 RAG 시스템을 OpenAI text-embedding-3-large로 구축했습니다. 법률 문서 특성상 한국어 판례문, 법령, 계약서가 주 데이터였는데, 오픈 베타 기간 중 사용자 피드백의 35%가 "관련 판례를 못 찾아온다"는 내용이었습니다.

내부 분석 결과, 한국어 법률 용어("손해배상책임" vs "손해를 배상할 책임이 있다" 등 표현 변형)에서 임베딩 유사도가 낮게 측정되는 것이 원인이었습니다. A사는 6주에 걸쳐 전체 문서를 `BAAI/bge-m3` 오픈소스 모델로 재임베딩하고, 한국 법률 도메인 문서로 파인튜닝했습니다. 결과적으로 Recall@5가 58%에서 79%로 향상되었고, 사용자 만족도 점수가 3.2→4.1 (5점 만점)로 개선됐습니다.

### 사례 2: 글로벌 SaaS 기업의 다국어 고객지원 RAG (익명, 2025~2026년)

한국·일본·독일·스페인 4개국에서 고객지원 RAG를 운영하는 B사는 처음에 국가별로 다른 임베딩 모델을 사용하는 복잡한 구조로 시작했습니다. 한국어는 Cohere Multilingual, 영어는 OpenAI, 일본어는 오픈소스 등.

6개월 운영 후 유지보수 복잡도와 언어 간 크로스링구얼 검색 부재(영어 문서를 한국어 쿼리로 찾기 어려움)가 문제가 됐습니다. B사는 전체 4개국 문서를 Cohere Embed v4 단일 모델로 통합했습니다. 운영 복잡도가 75% 감소하고, 크로스링구얼 검색 정확도가 새롭게 도입되면서 국가 간 지식 공유율이 40% 향상됐습니다.

### 사례 3: 국내 대형 금융사의 내규 검색 RAG (익명, 2026년)

국내 상위 5대 금융그룹 계열사 C사는 수만 건의 내부 규정, 컴플라이언스 문서를 임직원이 검색할 수 있는 RAG를 구축했습니다. 첫 아키텍처 설계 때 OpenAI API를 검토했지만, 정보보안팀에서 "고객 정보와 연관된 내규 문서가 외부 API 서버로 전송될 수 없다"는 결정을 내렸습니다.

결국 C사는 Cohere Private Deployment로 자사 클라우드 내부에 임베딩 모델을 배포하고, 데이터가 완전히 사내 환경에서만 처리되도록 구성했습니다. 초기 구축 비용은 퍼블릭 API 대비 높았지만, 연간 규제 준수 리스크 관리 비용과 잠재적 데이터 유출 리스크를 고려하면 합리적인 선택이었습니다.

## ⚠️ RAG 임베딩 모델 선택에서 빠지기 쉬운 5가지 함정

### 함정 1: MTEB 1위 모델이 내 도메인에서도 1위라는 착각
범용 벤치마크 점수와 도메인 특화 성능은 다릅니다. 반드시 자사 데이터로 직접 평가하세요.

### 함정 2: 초기 임베딩 비용만 보고 쿼리 비용 무시
문서 임베딩은 1회성이지만, 쿼리 임베딩은 매일 발생합니다. 트래픽이 많을수록 쿼리 비용이 초기 임베딩 비용을 수십 배 초과합니다. 월간 예상 쿼리 수를 기반으로 TCO(총 소유 비용)를 계산하세요.

### 함정 3: 고차원 = 고성능이라는 오해
OpenAI text-embedding-3-large의 3,072차원이 Cohere의 1,024차원보다 항상 좋은 건 아닙니다. 차원이 높을수록 벡터 DB 저장·검색 비용이 증가하고, 일부 태스크에서는 과적합(overfitting) 효과로 오히려 성능이 낮아질 수 있습니다. MRL을 활용한 차원 축소를 적극 고려하세요.

### 함정 4: 청킹 전략 없이 임베딩 모델만 바꾸는 실수
임베딩 품질은 청킹 방식(고정 크기, 의미 단위, 문단 기반 등)과 긴밀하게 연결됩니다. 임베딩 모델을 바꿔도 청킹이 잘못되면 효과가 반감됩니다. 임베딩 모델과 청킹 전략을 함께 최적화하세요.

### 함정 5: 한번 선택하면 영원히 쓴다는 고정관념
문서 양이 소규모일 때는 비싼 large 모델, 규모가 커지면 small + 파인튜닝 조합으로 전환하는 것도 전략입니다. 모델 교체 비용을 예산에 미리 반영해두면 더 유연한 운영이 가능합니다.

## ❓ 자주 묻는 질문

**Q1: OpenAI embedding이랑 Cohere embedding 차이가 뭔가요?**

A1: OpenAI의 text-embedding-3-large와 Cohere의 Embed v4는 모두 최상위 임베딩 모델이지만 설계 철학이 다릅니다. OpenAI 임베딩은 범용 영어 텍스트에 최적화되어 있고 OpenAI 생태계(ChatGPT, GPT-4o 등)와 seamless하게 연동됩니다. 반면 Cohere Embed v4는 100개 이상 언어 지원, 이미지+텍스트 멀티모달 임베딩, 128,000 토큰 컨텍스트, 기업 온프레미스 배포 지원이 강점입니다. RAG 프로젝트에서 영어 단일 언어·빠른 프로토타이핑이 목적이면 OpenAI, 한국어 포함 다국어 처리나 대규모 기업 환경이라면 Cohere가 우세한 경우가 많습니다. 단순히 "유명한 것"을 고르면 비용 낭비와 검색 품질 저하가 동시에 옵니다.

**Q2: RAG 임베딩 모델 바꾸면 벡터 DB 전체 다시 만들어야 하나요?**

A2: 네, 맞습니다. 임베딩 모델을 교체하면 벡터 공간 자체가 달라지기 때문에 기존에 생성해 둔 벡터 인덱스 전체를 다시 임베딩해야 합니다. 예를 들어 OpenAI text-embedding-3-small로 만든 벡터 DB에 Cohere Embed v4로 쿼리를 날리면, 전혀 다른 벡터 공간에서 유사도 계산을 하는 셈이라 검색 결과가 엉망이 됩니다. 따라서 프로젝트 초기에 임베딩 모델을 신중하게 선택하는 것이 중요하며, 추후 변경을 고려한다면 모델 교체 비용(재임베딩 연산 비용 + 시간)을 반드시 예산에 포함해야 합니다. 소규모 파일럿에서 A/B 테스트 후 확정하는 전략을 강하게 권장합니다.

**Q3: OpenAI embedding 가격, Cohere랑 비교하면 어떻게 되나요?**

A3: 2026년 4월 기준, OpenAI text-embedding-3-small은 $0.02/1M 토큰, text-embedding-3-large는 $0.13/1M 토큰입니다. Cohere Embed v4는 $0.10/1M 토큰 수준으로, 사실 large 모델과 유사한 가격대입니다. 단, Cohere는 Trial(무료 체험) 플랜으로 월 1,000회 API 호출까지 무료 테스트가 가능하고, 기업용 Private Deployment(전용 클러스터)는 별도 견적이 필요합니다. 대규모 문서(1억 토큰 이상)를 초기 임베딩하는 경우, small vs large 모델 선택만으로도 6.5배 비용 차이가 나므로 요구사항에 맞는 모델 티어를 먼저 확정하는 것이 중요합니다. 쿼리 볼륨이 높은 서비스라면 반드시 월간 TCO를 시뮬레이션해보세요.

**Q4: 한국어 RAG 만들 때 어떤 임베딩 모델 써야 하나요?**

A4: 한국어 RAG 구축 시 임베딩 모델 선택은 영어 기준과 완전히 달라집니다. OpenAI text-embedding-3 시리즈는 한국어도 지원하지만, 한국어 전용 벤치마크(KoSTS, KLUE-STS)에서 성능이 영어 대비 최대 15~20% 낮게 나오는 경우가 있습니다. Cohere Embed v4 멀티링구얼 모델은 한국어를 포함한 100개 이상 언어를 전용 학습 데이터로 커버합니다. 오픈소스 대안으로는 BAAI/bge-m3, intfloat/multilingual-e5-large-instruct 등도 한국어 RAG에서 경쟁력이 있으며, 자체 서버에서 운영하면 API 비용이 0입니다. 예산과 인프라 역량을 함께 고려하여 최종 선택 전 반드시 자사 도메인 문서로 소규모 평가를 해보세요.

**Q5: 임베딩 모델 성능 어떻게 테스트하면 되나요? 벤치마크만 믿어도 되나요?**

A5: MTEB(Massive Text Embedding Benchmark)가 현재 가장 신뢰받는 임베딩 모델 벤치마크지만, 범용 벤치마크 점수가 여러분의 도메인에서도 동일하게 적용된다는 보장은 없습니다. 의료·법률·금융처럼 전문 용어가 많은 도메인에서는 일반 벤치마크 1위 모델이 특화 모델보다 낮은 검색 정밀도(Precision@k)를 보이는 경우도 있습니다. 실전 테스트 방법은 ①자사 문서 200~500개로 샘플 코퍼스 구성 → ②대표 쿼리 50~100개 준비 → ③각 모델로 임베딩 후 Recall@5, MRR 직접 측정 → ④비용·속도까지 종합해 최종 결정 순서를 권장합니다. RAGAS 라이브러리를 활용하면 자동 평가 파이프라인을 쉽게 구축할 수 있습니다.

## 마무리: 임베딩 모델 선택은 프로젝트의 첫 번째 아키텍처 결정

"어차피 나중에 바꾸면 되지"라는 생각으로 임베딩 모델을 선택하면, 프로덕션 이후 엄청난 재작업 비용을 치르게 됩니다. 반대로 "무조건 비싼 거, 유명한 거"를 선택해도 한국어 RAG에서 검색 품질이 기대에 못 미치거나, 불필요한 비용이 발생할 수 있습니다.

지금 여러분의 RAG 프로젝트에 맞는 임베딩 모델을 선택하는 기준은 딱 4가지입니다.

1. **순수 검색 성능** — 도메인 데이터로 직접 측정
2. **비용 구조** — 초기 + 쿼리 볼륨 기반 TCO 계산
3. **언어 지원** — 한국어/다국어가 있다면 전용 모델 필수
4. **보안·컴플라이언스** — 초기 아키텍처 설계 단계에서 확정

임베딩 모델 비교는 스펙 시트 비교가 아니라, 여러분 프로젝트의 요구사항을 먼저 정의하고 거기에 맞는 모델을 역방향으로 찾아가는 과정입니다.

지금 여러분은 어떤 환경에서 RAG를 구축하고 계신가요? 한국어 도메인인지, 보안 요구사항이 있는지, 쿼리 볼륨은 어느 정도인지 댓글로 알려주시면, 상황별 최적 임베딩 모델 조합을 더 구체적으로 안내해 드리겠습니다. 특히 "한국어 + 대규모 기업 환경"이라는 조건이면 선택지가 생각보다 좁아집니다 — 이 케이스에 대해 다음 글에서 더 깊이 파고들 예정이니 기대해 주세요.
